## 2. マルチテナント設計

### 2-1. テナント = 1企業 = 1スプレッドシート

```
Firestore（メタデータ）
└── tenants/
    └── {tenant_id}/
        ├── name: "株式会社〇〇"
        ├── spreadsheet_id: "1aBcDeFgHiJkLmNoPqRsTuVwXyZ..."
        ├── created_at: timestamp
        ├── plan: "trial"
        └── semantic_markdown: "# 経営方針 v1.0\n..."
```

- テナント新規作成時、Google Sheets APIで**テンプレートスプレッドシートを自動コピー**し、`spreadsheet_id`をFirestoreに保存
- 全てのSheets操作は `tenant_id` → `spreadsheet_id` の参照を経由し、テナント間のデータは完全に分離

### 2-2. テンプレートスプレッドシート構成

1テナントにつき1つのスプレッドシートを発行。以下の4シートで構成。

| シート名 | 役割 |
|---|---|
| `t_kpi_logs` | KPI実績・目標・個別体温の月次ログ |
| `t_survey_responses` | 匿名アンケートの回答蓄積 |
| `t_ai_insights` | AI診断結果の出力先 |
| `m_departments` | 部署マスタ（部署名・人数・部署ID） |

## 7. スコアリングロジック

### 7-1. KPI個別スコア

```
IF 目標値あり:
  評価値 = 実績値 ÷ 目標値（達成率）

ELSE IF 前月データあり:
  推移率 = (当月実績 - 前月実績) ÷ 前月実績
  IF 推移率 < 0:
    評価値 = 1 + 推移率 × |推移率|^0.5   // マイナス時は指数関数的ペナルティ（1.5乗）
  ELSE:
    評価値 = 1 + 推移率

ELSE:
  評価対象外（スコアなし）

個別スコア = 評価値 × 個別体温係数(pulse_coefficient)
```

### 7-2. 個別体温係数の算出

各KPIに関連するアンケート集計結果から算出。

```
部署のアンケート平均スコア（1-5）:
  ☀️ (1.2): 平均 ≥ 3.8
  ☁️ (1.0): 平均 2.5 〜 3.8
  ☔️ (0.5): 平均 < 2.5
```

### 7-3. 全体体温（ダッシュボードのメインアイコン）

```
全体体温 = 全KPIの個別体温係数の平均値

  ☀️: 全体体温 ≥ 1.1
  ☁️: 全体体温 0.7 〜 1.1
  ☔️: 全体体温 < 0.7
```

### 7-4. 推移矢印

```
当月の全体体温 vs 前月の全体体温:
  ↑: +0.1以上の改善
  ↓: -0.1以上の悪化
  →: 上記以外（横ばい）
```

### 7-5. オーバーヒート検知

```
IF KPIスコア上位（達成率 ≥ 100%）AND 体温係数 = 0.5（☔️）:
  → 「オーバーヒート」フラグ
  → AI診断に「数字は出ているが現場は限界。離職爆発リスク」を最優先で含める
```

### 7-6. マトリックス用・一人当たり生産性スコア

部署別・プロダクト別の散布図で、縦軸に使用するスコア。
部署ごとに主要KPIが異なるため、達成率ベースで正規化して比較可能にする。

```
【部署別の場合】
  その部署が主担当のKPI群の達成率平均を算出

  部署生産性スコア = 主担当KPI達成率の平均 × 体温係数

  ※ KPI目標自体が「現在の人数（リソース）に対する経営からの期待値」であるという前提に立ち、目標達成率をベースに標準化（インデックス化）することで、営業からバックオフィスまで全球の部署を同列のY軸（生産性）で比較可能にします。
  ※ 主担当KPIが複数ある場合（例：営業部 → MRR, 商談数, 成約率）は平均を取る
  ※ 【重要思想】すべての部署（人事・経理等のバックオフィス含む）は何らかのKPI（コスト削減、採用工数、NPS等）を持ちます。人数の増大（投資）は、その部署の「KPIを最大化（または最小化）」する目的にのみ紐づいて判断されるべきであるため、例外なく全球の部署をマトリックス上に可視化して生産性を問います。

【プロダクト別の場合】
  プロダクト生産性スコア = (プロダクトMRR × 体温係数) ÷ 担当人数

  ※ プロダクト別表示はV2（プロダクトマスタ追加後）で実装
```

### 7-7. KPI別アンケート紐付け

アンケートの `related_kpi` フィールドを使い、各KPIに関連する自由記述を抽出。
AI診断時に「このKPIの担当部署はこう言っている」としてコンテキストに含める。

```
1. t_survey_responses から related_kpi = "MRR" のレコードを抽出
2. free_comment をClaudeに渡し、3件以内の要約を生成
3. 要約は t_ai_insights に kpi_name 付きで保存、ダッシュボードのKPI詳細画面に表示
```

## 8. AI診断（140文字）生成仕様

### 8-1. 入力コンテキスト

Claude APIへ送るプロンプトに以下を全て含める。

```
1. セマンティックレイヤー（Markdown本文） ← Firestoreから取得
2. 当月のKPIログ（t_kpi_logs）
3. 前月のKPIログ（比較用）
4. 当月のアンケート集計結果（部署別平均・ボトルネック集計・自由記述サマリー）
5. 算出済みのスコア・天気・矢印
6. 対象の階層ロール（Executive / Admin / Manager / Player）
```

### 8-2. 出力制約

| 制約 | 内容 |
|---|---|
| 文字数 | 140文字以内（厳守） |
| 口調 | 断定調。曖昧な表現を避ける |
| トーン制御 | ロール別に切替（下記参照） |
| 禁止事項 | 個人名の言及、曖昧な楽観表現 |

### 8-3. 階層別トーン

| ロール | 視点 | トーン | 例 |
|---|---|---|---|
| Executive | ROI・リソース配分・事業判断 | 冷徹な軍師 | 「A事業の先行指標が20%低下。来月確実に数字が落ちる。B事業の余剰人員をAへ配置転換すべき。」 |
| Admin | プロセスの詰まり・離職予兆・組織構造 | 構造的・警告的 | 「マーケのリード質低下が営業の疲弊を招いている。業務フロー見直しとケアを同時に。」 |
| Manager | チーム士気・ボトルネック言語化 | コーチング・伴走 | 「メンバーは手法に迷っている。数字を詰めるより方針の再言語化が今の優先事項。」 |
| Player | 心理的安全性・不条理の代弁 | 共感・翻訳 | 「あなたが感じている製品の課題は他3名も同回答。個人の問題ではなく仕組みの問題として報告済み。」 |

### 8-4. 部署間360度フィードバックの統合

- アンケートの「他部署フィードバック（cross_dept_feedback）」をAIがクラスタリング
- 「マーケ→営業」「営業→開発」等の方向性付きで診断に組み込み
- ポジティブな相関を優先的に抽出（告げ口ツール化の防止）
- 部署名は出すが個人名は絶対に出さない

## 9. セマンティックレイヤー仕様

### 9-1. 概要

経営方針・フェーズ・KPIの解釈をMarkdownで記述し、AI診断時のコンテキストとして使用。Notionなどで別途作成したMarkdownをコピー＆ペーストで投入可能。

### 9-2. 保存先

Firestoreの `tenants/{tenant_id}/semantic_markdown` フィールド（string型）。

### 9-3. 推奨テンプレート

```markdown
# 経営方針 v1.0（2026年1月〜）

## 組織の現在地
- フェーズ: 垂直立ち上げ
- 現場に求める空気感: 失敗を恐れない突破力

## KPIの解釈ガイド
- 商談数: 今は量を最重視。質の精査はフェーズ2で。
- 解約率: 3%以下なら許容。プロダクト改善に集中。
- MRR: 月次20%成長がNorth Star。

## 数字と体温の関係
- ☀️なのに☔️（オーバーヒート）: 今は踏ん張り時と捉える。ただし2ヶ月連続なら即介入。
- ☔️なのに☀️: 仕込み期間として許容。North Starが伸びていれば継続。

## 地雷ワード（AIが検知したら最優先で報告）
- 「辞めたい」「転職」「意味がない」「誰も聞いてくれない」
```

### 9-4. UI

- 管理設定画面内にMarkdownテキストエリア（等幅フォント）
- プレビュー機能（任意。MVP後でも可）
- 「保存」ボタンで Firestore に即反映
- 更新日時を `semantic_updated_at` に記録

## 11. 主要フロー

### 11-1. テナント初期設定フロー

```
管理者がGoogleログイン
  → 新規ユーザー判定
  → 企業名入力
  → 部署登録（名称・人数、複数追加）
  → KPI定義登録（名称・単位・目標値、最大10個）
  → セマンティックレイヤー入力（Markdown、スキップ可）
  → [システム] テンプレートスプレッドシートを自動コピー
  → [システム] m_departments にマスタ書き込み
  → ダッシュボードへ遷移
```

### 11-2. 月次運用フロー

```
【月初〜月中】
  1. 管理者: 前月のKPI実績をKPI入力画面から入力 → t_kpi_logs に追記
  2. 管理者: アンケートURL管理画面から部署別URLを現場に配布
  3. 現場: スマホ等からアンケートに回答 → t_survey_responses に蓄積
  4. 管理者: 回答状況（回答数/所属人数）をアンケートURL管理画面で確認

【集計タイミング（管理者が手動実行）】
  5. 管理者: 「集計を実行する」ボタンを押下
  6. [システム] スコアリングロジック実行
     - t_kpi_logs から当月・前月データ取得
     - t_survey_responses から当月回答を集計
     - 体温係数・天気・矢印を算出
  7. [システム] Claude APIに診断プロンプト送信
     - セマンティックレイヤー + KPIデータ + アンケート集計 を入力
     - 4階層 × 各部署 + 全社分の診断を生成
  8. [システム] t_ai_insights に結果を書き込み
  9. [システム] アンケート回答数をリセット（次月用）
  10. ダッシュボードに最新の天気・診断・KPIが反映
```

### 11-3. セマンティックレイヤー更新フロー

```
管理者: 設定画面でMarkdownを編集・保存
  → Firestoreの semantic_markdown を上書き
  → 次回の「集計を実行する」時にAIが新方針を参照
```
